SpringBoot 4.0升级

mengnankkzhou2026-01-022026-01-02

背景

最近Spring AI 发布了2.0.0的测试版本，标志着SrpingAi社区开始维护1.x版本和2.x版本。然后发现里面最大的一个区别就是Spring boot版本的升级，由之前的3.x版本升级到了4.x版本

我们首先来看下Spring boot的版本升级，为什么会再去开一个版本分支去维护呢？

Springboot4.0

Spring Boot 4.0.0 的发布，不仅仅是一个版本号的简单的递增，它标志着 Spring 生态系统正式迈入了基于 Spring Framework 7.0、Java 17/25 以及 Jakarta EE 11 的全新纪元。对于当前运行在 Spring Boot 3.5.8 的企业级应用而言，向 4.0.0 的迁移不仅是一次技术栈的升级，更是一次对底层基础设施、开发范式以及运维观测体系的全面现代化重构 。

新旧版本的对比：

核心组件 Spring Boot 3.5.8 (当前状态) Spring Boot 4.0.0 (目标状态) 架构影响分析
Spring Framework 6.2.x 7.0.x 引入了全新的 Bean 注册机制、API 版本控制及观测性增强 。
Java Baseline Java 17 Java 17 (推荐 Java 25) 深度集成虚拟线程、ZGC 优化及语言新特性（如 Records, Pattern Matching） 。
Jakarta EE EE 10 EE 11 Servlet 6.1 带来了 I/O 模型的变革，直接导致了 Undertow 支持的暂时移除 。
Kotlin 1.9 / 2.0 2.2 K2 编译器的原生支持，编译速度与空安全互操作性的显著提升 。
Observability Micrometer Tracing (Brave/Otel) OpenTelemetry (OTLP) 从库优先转向协议优先，原生集成 OTLP Exporter，简化云原生监控栈 。
Build Tool Gradle 8.x Gradle 9.x 构建性能优化与配置缓存的深度集成 。

那为什么升级？

升级原因

2.1 Java 版本的战略选择：为何是 25？

虽然 Spring Boot 4.0 保持了 Java 17 作为最低支持版本，以兼顾企业现有的 LTS 策略，但其设计核心显然是围绕 Java 25 展开的 。Java 25 预计将于 2025 年 9 月发布，Spring Boot 4.0 对其进行了前瞻性的适配。

2.1.1 虚拟线程的全面释放

在 Spring Boot 3.x 中，虚拟线程（Project Loom）的支持尚处于孵化和适配阶段。而在 4.0 版本中，虚拟线程已成为一等公民。当配置 spring.threads.virtual.enabled=true 时，Spring Boot 不仅仅是简单地将 Tomcat 的线程池替换为虚拟线程执行器，而是对整个调用链进行了深度的非阻塞改造 。

  • Web 容器层：Tomcat 11 和 Jetty 12 的集成完全适配了虚拟线程模型，这意味着单个应用实例可以轻松处理数万甚至数十万的并发连接，而无需像 Reactive 栈那样编写复杂的异步回调代码。
  • 数据访问层：Spring Data 2025.1 对 JDBC 驱动进行了适配，使得传统的阻塞式数据库调用在虚拟线程中运行时，能够自动挂起（unmount）虚拟线程，释放载体线程（Carrier Thread）去处理其他任务。这使得 “Thread-per-request” 模型在 I/O 密集型场景下重新焕发了生机，性能直逼 WebFlux 。

2.1.2 语言特性的深度利用

Spring Framework 7.0 内部大量采用了 Java 17+ 的新特性。例如，利用 Records 来优化配置属性类的定义，利用 Sealed Classes 来增强框架内部类型体系的严谨性。对于开发者而言，这意味着在编写业务代码时，可以毫无顾虑地使用 Switch Expressions、Text Blocks 以及 Pattern Matching，因为框架层已经完全消除了对旧版字节码的兼容包袱。

2.2 Jakarta EE 11：标准化的代价与收益

Spring Boot 4.0 是首个基于 Jakarta EE 11 构建的主流框架版本。这一升级带来的最大震动在于 Servlet 6.1 规范的实施 。

2.2.1 Servlet 6.1 与 I/O 模型变革

Servlet 6.1 引入了更加现代化的 I/O 处理机制，旨在更好地支持 HTTP/2 和未来的 HTTP/3 协议。它强制要求容器实现更高效的缓冲区管理和非阻塞 I/O。

  • Undertow 的移除：这一规范的升级直接导致了一个重大的生态变动——Undertow 支持被移除 。由于 Undertow 社区在 Spring Boot 4.0 发布时尚未完成对 Servlet 6.1 的完整兼容，Spring 团队做出了艰难的决定，暂时在 4.0 版本中剔除 Undertow。对于习惯使用 Undertow 作为高性能 Web 容器的用户，升级到 4.0 意味着必须迁移回 Tomcat（默认）或 Jetty。这是一项需要谨慎评估性能影响的变更。

2.2.2 JPA 3.2 与 Hibernate 7

Jakarta Persistence 3.2 带来了更强的查询表达能力。Spring Boot 4.0 默认集成了 Hibernate 7.1 。Hibernate 7 在性能上进行了大幅优化，特别是在复杂关联查询的 SQL 生成策略上。此外，Hibernate 7 对 Jakarta Validation 3.1 的支持也更加完善，特别是在校验 Java Records 类型时，能够直接读取构造函数参数上的注解进行校验，减少了样板代码

新的升级

  1. 新的 @RequestMapping 注解增加了一个 version 属性。

框架支持多种版本解析策略，且均可通过 application.properties 进行全局配置：

  • Path Variable: 解析 URL 中的版本段。
  • Header: 如 X-API-Version: 1。
  • Parameter: 如 ?version=1。
  • Media Type: 如 Accept: application/vnd.mycompany.v1+json。

这种原生支持不仅减少了样板代码，还使得 Swagger/OpenAPI 的文档生成能够自动识别多版本接口，极大地提升了 API 治理的规范性。

3.空指针的处理

Spring Boot 4.0 全面采用了 JSpecify 规范（JSR-305 的现代继任者）来标注 API 的空安全性 。

  • @NullMarked：这是一个包级注解。Spring 团队已经将几乎所有的核心包都标记为 @NullMarked。这意味着，除非特别标注，否则所有方法的参数和返回值都默认视为 非空（Non-null）。
  • @Nullable：对于确实允许为 null 的场景，框架会显式使用 JSpecify 的 @Nullable 进行标注。
